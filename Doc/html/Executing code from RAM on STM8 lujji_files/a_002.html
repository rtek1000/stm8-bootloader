Because the .map file shows e.g. RAM0 in flash at 0x81E6. If my understanding is correct, how do you ensure that absolute jumps/calls within RAM are not used by SDCC?

                2) and if absolute jumps or calls within RAM are required for some reason, how about this (different) procedure:

                a) link for RAM (-Wl-bRAM0=0x00a0) --> absolute addresses are ok, but in hexfile code is in RAM and is lost after reset (already verified)

                b) in hexfile move code segment from RAM to flash address --> is not lost after reset, but located at wrong address. I plan to add this "move" feature to https://github.com/gicking/... within the next few days

                c) copy to same RAM address as above just like you already do, but with an additional __at (0x00a0) for f_ram[] to ensure the correct address.

                Optionally one could also reserve a buffer of sufficient size in RAM @ 0x00a0 (or wherever) via linker and only point *f_ram to that address in the main program.

                The main issue I see is how to pass the parameters like link address and code length between the different tools/steps.

                I understand that this is not minimalist anymore (sorry for that!). But is my understanding correct and, if yes, any idea how to perform this in a fool-proof way?

                    About a year ago I bought a batch of 20 of those little STM8 boards, and apart from making one of them blink I haven't done anything further. I had some vague plans to wire them together in a network to experiment with artificial life, with each node self-modifying its code through mutation, neighbours peeking and poking each other's memory etc. Coming across your article has moved that along, because executing native code (from RAM) would be so much simpler than having to code some kind of VM. In this case, the read only flash would contain network I/O routines and a supervisor that is invoked by the watchdog timer if the 'live' code mutates sufficiently to stop working at some defined level...but otherwise the running code in RAM would be left to go rogue.

                    Got my programmer connected to one port, and serial to another, time to get to work! Thanks for this article, and your other STM8 tutorials.
